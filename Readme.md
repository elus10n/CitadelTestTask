# Configurable Data Extractor

Программный комплекс Configurable Data Extractor представляет собой решение тестового задания на стажировку, предназначенное для автоматизированного извлечения и статистического анализа данных телеметрии из неструктурированных текстовых источников.

## Назначение и функциональные возможности

Основная задача комплекса это сбор данных от распределенных сенсоров и поиск экстремальных (минимальных и максимальных) значений по заданным правилам с учетом нормализации единиц измерения.

Ключевые характеристики системы:
* **Конфигурируемая логика извлечения**: параметры поиска и типы данных (логические значения, числовые показатели, скорость передачи данных) задаются в формате JSON.
* **Параллельная обработка**: архитектура системы поддерживает асинхронное чтение и парсинг файлов, что позволяет сократить время обработки больших массивов данных.
* **Отказоустойчивость**: механизм обработки исключений позволяет продолжать работу при обнаружении некорректных записей в исходных файлах, фиксируя ошибки в отдельном потоке логирования.
* **Кроссплатформенная сборка**: проект использует систему автоматизации сборки CMake и поставляется с конфигурацией для развертывания в изолированных контейнерах (Docker).

## Структура проекта

Репозиторий имеет модульную структуру, разделенную на исходный код, заголовочные файлы и вспомогательные инструменты:

* **`include/`** внешние зависимости и общие заголовочные файлы.
    * `nlohmann/` библиотека для работы с JSON.
    * `models.h` описание основных структур данных системы.
* **`source/`** реализация бизнес-логики, разделенная по компонентам:
    * `Parser/` модуль обработки конфигурационного файла.
    * `Worker/` движок многопоточного парсинга текстовых данных.
    * `Aggregator/` логика статистической обработки и поиска экстремумов.
    * `Orchestrator/` координатор жизненного цикла приложения.
    * `Logger/` реализация системы обратных вызовов для логирования.
* **`example/`** демонстрационные материалы:
    * `data/` директория с входными текстовыми файлами (изначально те, что в ТЗ).
    * `example.json` пример конфигурации правил извлечения.
    * `fillScript.py` Python-скрипт для генерации тестовых наборов данных.
* **`main.cpp`** точка входа в приложение.
* **`CMakeLists.txt`**
* **`Dockerfile`**
* **`entrypoint.sh`**  сценарий автоматического запуска тестов и демонстрационного примера внутри контейнера.

## Сборка и запуск

### Системные требования
* Компилятор с поддержкой **C++17**
* **CMake** версии 3.14 или выше
* **Python 3** (для работы скрипта генерации данных)

### Использование Docker (рекомендуемый способ)
Проект настроен для автоматической сборки и запуска тестов внутри контейнера. Это гарантирует идентичность окружения.

```bash
# Сборка образа
sudo docker build -t citadel-analyzer .

# Запуск контейнера
# По умолчанию выполняется сценарий entrypoint.sh: 
# запуск unit-тестов, обработка примера и стресс-тест.
sudo docker run --rm citadel-analyzer
```

### Локальная сборка

Для компиляции проекта непосредственно в операционной системе необходимо выполнить следующие шаги:

#### 1. Системные требования
* Компилятор с поддержкой стандарта **C++17**
* Система автоматизации сборки **CMake** версии 3.14 или выше.

#### 2. Подготовка директории
Сборку рекомендуется производить в отдельном каталоге для сохранения чистоты исходного кода:
```bash
mkdir build && cd build
```

#### 3. Конфигурация и компиляция
```bash
# Генерация файлов сборки
cmake .. -DCMAKE_BUILD_TYPE=Release

# Сборка исполняемых файлов
cmake --build . --config Release
```
#### 4. Запуск
После успешного завершения сборки исполняемый файл main_app будет доступен в директории build. Запуск осуществляется с передачей трех обязательных аргументов:

##### Linux
```bash
./main_app <путь_к_config.json> <путь_к_директории_с_данными> <путь_к_отчету.txt>
```
##### Windows
```bash
Release/main_app.exe <путь_к_config.json> <путь_к_директории_с_данными> <путь_к_отчету.txt>
```

## Сборка и запуск
### Модульные тесты
В состав программного комплекса входят модульные тесты, реализованные с использованием фреймворка `Google Test`. Тесты покрывают логику парсинга конфигурации, извлечения данных и корректность работы агрегатора.

Запуск тестов (после локальной сборки)
#### Linux
```bash
./unit_tests
```
#### Windows
```bash
Release/unit_tests.exe
```

### Генерация тестовых данных
Для проведения нагрузочного тестирования и проверки работы с большим количеством файлов предусмотрен скрипт `fillScript.py` (Python 3). Скрипт генерирует набор файлов со случайными показаниями датчиков, включая некорректные записи для проверки отказоустойчивости системы.
```bash
python3 example/fillScript.py
```

## Формат выходных данных
Результатом работы программы является текстовый отчет. Для каждого сенсора и применимого к нему правила в отчете фиксируются:

* **Минимальное и максимальное значения**.
* **Пути к файлам**, в которых данные значения были обнаружены.
* **Нормализованные величины** (с учетом конвертации единиц измерения скорости в биты).
